## How to work

- **Always read the relevant doc before implementing** — these specs exist to prevent rework
- **Migration discipline**: any schema change goes in `supabase/migrations/000N_descriptive_name.sql`. Never edit existing migrations. Regenerate types after each.
- **RLS is non-negotiable**: every new table needs RLS enabled and policies written before it ships. The first migration only has read policies stubbed — you'll need to write insert/update/delete policies per module as you build (lines marked `NOTE` in `0001_initial_schema.sql`).
- **Server-side validation always**: client zod is for UX, server zod is for security.
- **Don't add new dependencies without asking** — the `package.json` was scoped carefully.
- **Tests where it matters**: business logic (bulk import matching, future scoring engine) gets Vitest unit tests. Happy paths get Playwright E2E.
- **Indonesian for UI copy, English for code/comments**: the product is for Indonesian users.

## When you're stuck

- If a doc is ambiguous, ask — don't guess
- If a spec contradicts itself, ask — likely an oversight in foundation docs that needs to be fixed there first
- If you need to deviate from the stack/structure, propose the change with reasoning before doing it

1. **Buka Claude Code di folder `milestone-tracker-app/`** (bukan parent folder). Claude Code auto-detect docs di situ.
2. **Pakai model paling kuat (Opus)** untuk session pertama — ada banyak konteks yang harus diserap.
